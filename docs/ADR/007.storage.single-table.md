title: ADR-007 AI 供应商管理采用单表 + GROUP BY 结构

## Context

AI 供应商管理需要持久化存储。设计候选：

- 两张表：`ai_providers`（供应商）+ `ai_models`（模型），外键关联
- 一张表：`ai_models`，供应商信息通过 `GROUP BY base_url, provider_name` 推导

约束：

- API Key 是 per model 级别，不是 per provider
- 供应商名称、base_url 等信息对同一供应商下所有模型相同
- 数据量不大（几十个供应商、几百个模型）
- 不想维护两表之间的外键关系和同步逻辑

## Decision

采用单表 `ai_models`，供应商通过 GROUP BY 推导。

Key 设计决策：

- `api_key` 字段在模型级别（per model）
- `base_url + model_id` 唯一索引防重复
- 供应商级操作（改名、删除）通过 `WHERE base_url = ?` 批量更新/删除
- `meta` JSON 字段存可变/扩展数据

## Consequences

优点：

- 结构简单，一张表搞定
- 无外键同步问题
- Key per model，灵活（同一供应商不同模型可以用不同 key）

缺点：

- 同一供应商的 name/endpoint/key 在每行重复存储（数据量小可接受）
- GROUP BY 查询比直接查 provider 表略慢（数据量小无感知）
