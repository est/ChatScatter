## Context

LLM 推理时，KV cache 是按 token 序列前缀命中的。在树结构下：

- 主线有一条固定的前缀路径（根到最近的摘要点）
- 每个分支有自己独立的消息序列

如果每个分支的上下文 = 主线前缀 + 分支消息，那么：
- 同一分支内连续对话：KV cache 完全命中（前缀不变）
- 切换到另一分支：主线前缀部分命中，分支部分 miss
- 回到主线：主线前缀完全命中

## Decision

不需要额外实现。树结构 + focus 指针天然适配 KV cache：

1. 主线前缀在所有分支间共享 → 高命中率
2. 每个分支的消息序列独立 → 切换分支时只有分支部分 miss
3. 分支摘要后，主线前缀变短 → 更多空间给新内容

实际优化手段：
- 上下文构建时，主线消息放前面，分支消息放后面
- 摘要替换原始消息后，前缀缩短，cache 命中率提升
- Provider 侧的 prompt caching（如 Anthropic cache_control）可以主动标记主线前缀

## Consequences

优点：

- 零额外代码成本，架构设计自然命中
- 分支越多，主线前缀越稳定，cache 命中率越高
- 摘要压缩 = 前缀缩短 = 省 token 省钱

缺点：

- 需要按"主线前缀在前、分支在后"的顺序组装 prompt
- 不同 provider 的 cache 策略不同，需要适配
