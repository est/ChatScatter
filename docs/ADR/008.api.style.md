## Context

需要统一 API 风格。

## Decision

### 路径规则

- 使用 `/api/{domain}/{verb}`，如 `/api/provider/list`、`/api/provider/probe`
- 查询参数放 query string，不把业务参数放进 URL path
- 抵制 RESTful 风格（web 日志无法归类）
- 不做版本号
- 不要搞单数、复数之类的自讨苦吃

### HTTP 方法

- `GET` 读
- `POST` 写入或提交命令

不用 `PUT`、`PATCH`、`DELETE`。

### 请求

如果期待 JSON返回，那么头 `Accept: application/json` 。方便服务器做一个端点多种输出。

给浏览器API的就用 cookie。给SDK用的API的走 `Authorization` 头。

### 响应

所有接口统一返回：

```json
{
  "data": {},
  "em": ""
}
```

- `data`：返回数据，失败时为 `null`
- `em`：error message。falsy 值表示无错误；有值字符串表示错误信息
- 如无特殊理由，一律返回 200

### 请求格式

请求体一律用 `{"data": ...}` 包裹，以便以后加字段有地方放。

## Consequences

- 日志按路径归类方便（`/api/provider/list` 而非各种 HTTP method + path param 组合）
- 前端统一判断 `em` 即可
