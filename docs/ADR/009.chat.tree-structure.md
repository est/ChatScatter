## Context

传统 AI 聊天是一维消息列表。但人脑思维是发散的：一个话题会分叉出子问题，子问题探索完会回到主线。

需要一种数据结构，能表达：
- 主线方向
- 从主线分叉出的支线探索
- 支线完成后回到主线
- 支线内部还可以再分叉

候选：
- 线性列表（传统聊天）
- 树结构（主线为根，支线为子树）
- 图结构（任意节点可连接任意节点）

## Decision

采用树结构。每个节点是一条消息或一个分支点。

概念模型：

```
[root]
  ├─ msg: "我想做一个聊天系统"
  ├─ msg: "先研究下数据结构"
  │   ├─ branch: "树 vs 图？"        ← 分支A
  │   │   ├─ msg: "树更简单"
  │   │   └─ msg: "结论：用树"
  │   └─ branch: "KV cache 影响？"   ← 分支B
  │       └─ msg: "每个分支独立 cache"
  ├─ summary: "数据结构：树 + focus 指针"  ← 分支A/B 压缩
  └─ msg: "好，开始设计 API"
```

先不做图。

## Rejected Alternatives

### 图结构

挑战了树的决定，结论是树仍然正确：

1. **交叉引用**：图能表达"A 分支结论影响 B 分支"。但这是引用关系，不是结构关系。节点内用 `refs: ["node-xxx"]` 字段即可，不需要把树变成图。保持树形导航不变，引用只是"可以跳转"。

2. **分支合并**：两条支线可能收敛到同一结论。但极少见，且手动在主线写一条总结引用两个分支的结论就够了。不值得为此引入图。

3. **人脑是图**：是的，但人用工具时需要约束。树是"有约束的图"，约束本身就是价值。没有约束的图 → 用户自己也忘了在干嘛。

4. **"之前聊过相关的"**：展开已摘要分支即可回溯，不需要图结构。

关键认知：**树 + 引用字段 > 图**。引用是数据，不是结构。

## Consequences

优点：

- 侧边栏天然映射为树形导航
- 分支可以独立折叠/展开，不影响主线
- 每个分支的上下文独立，可以单独 cache
- 结构简单，D1 一个 parent_id 字段就能表达
- root 作为锚点，提醒用户聚焦（工具性 > 闲聊性）

缺点：

- 不能直接在结构层面表达交叉引用（用字段解决）
- 树深度过深时导航变复杂（通过摘要压缩缓解）
