## Context

树结构解决了数据组织问题，但 LLM 只接受线性 token 序列。需要一个机制，把树的某个"切片"映射为线性上下文。

问题是：当用户在分支 B 里聊天时，系统应该给 LLM 看什么？

选项：
- 看整棵树的所有消息（上下文爆炸）
- 只看当前分支（丢失主线背景）
- 看从根到当前节点的路径（主干 + 当前分支）

## Decision

引入 focus 指针：一个指向树中某个节点的游标。

LLM 的上下文 = **从根到 focus 节点的路径上所有消息，沿途分支已被摘要替换。**

```
[root] ──msg──msg──[summary of branch A]──msg──[focus: here]
                       ↑
                  分支A 被压缩成一行摘要
```

切换 focus 时：
1. 离开分支 → 自动/手动压缩该分支为摘要
2. 进入分支 → 展开分支，加载分支内的完整消息
3. 回到主线 → 主线上下文不变，分支摘要保留

## Consequences

优点：

- LLM 上下文始终是线性的，不需要改 prompt 结构
- 分支切换 = focus 移动 + 上下文重建，逻辑清晰
- 主线不会被支线细节污染

缺点：

- 需要维护"从根到 focus 的路径"计算
- 摘要质量影响 LLM 对已压缩分支的理解（摘要策略见 ADR-011）
