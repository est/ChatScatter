## Context

ADR-009 确定了树结构，ADR-011 确定了分支摘要压缩。但没有回答：

1. 分支从哪里创建？
2. 谁来生成摘要？
3. 用户如何触发分叉？

候选方案：

- 用户手动创建分支（操作驱动）：需要额外 UI 操作，增加认知负担
- 系统根据 assistant 回复自动创建分支（结构驱动）：markdown 标题 = 分支入口
- 混合：系统建议，用户确认

摘要生成候选：

- 用户手动写摘要
- 单独调一次 LLM 生成摘要（额外 API 调用）
- assistant 在回复时 inline 嵌入摘要 metadata（无额外调用）

## Decision

### 分支由 assistant 的 Markdown 结构驱动

流程：

```
user 问问题
  → assistant 回复 markdown（含多个 ## 段落）
    → 每个 ## 自动成为候选分支
      → 用户点击某个 ## → 进入子分支
        → 子分支上下文 = 该段落 + 用户追问
          → assistant 继续回复（可再分叉）
```

不需要"创建分支"按钮。Markdown 标题就是分支入口。

### 摘要由 assistant inline 生成

assistant 在回复时自然嵌入结构化 metadata，不是额外生成：

```markdown
## 数据结构

讨论了树 vs 图。结论：树 + 引用字段 > 图。

<!-- branch-meta: { "topic": "数据结构", "summary": "树+refs>图", "depth": 1 } -->
```

- 无额外 API 调用
- 摘要质量 = assistant 输出质量
- metadata 随消息一起存储

### 候选分支的生命周期

用户深入了段落 A，段落 B/C/D 成为候选分支：
- 保留在树上，状态为"未探索"
- 不占上下文
- 用户可随时回来点击其他段落
- 如果用户再也没回来，就是"未走的路"，无害

## Consequences

优点：

- 分支创建零 UI 成本 — markdown 结构即分支
- 摘要零额外成本 — inline 生成
- 符合 markdown 阅读习惯 — 标题 = 导航
- assistant 的回复质量决定分支质量 — 用 prompt 引导即可

缺点：

- 依赖 assistant 输出结构化 markdown（需要 prompt 工程）
- metadata 格式需要约定（HTML comment / JSON frontmatter / YAML）
- 非 markdown 回复（纯代码、纯图片）不产生分支点
