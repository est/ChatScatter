## Context

ADR-009 确定了树结构。需要定具体的 D1 表结构。三个设计决策：

1. 用户消息和助手消息如何存储？
2. 节点 ID 用什么格式？
3. 树的父子关系如何表达？

## Decision

表结构：

- conversation_id TEXT NOT NULL,
- id              BLOB NOT NULL,       -- conv 内从1 开始自增，2-byte 
- parents         TEXT NOT NULL DEFAULT '',-- 祖先路径，如，2byte 一组 形成前追
- user_content    TEXT NOT NULL,
- user_meta       TEXT DEFAULT '{}',
- assistant_content TEXT NOT NULL DEFAULT '',
- assistant_meta  TEXT DEFAULT '{}',
- meta            TEXT DEFAULT '{}',
- created_at      TEXT NOT NULL,
- PRIMARY KEY (conversation_id, id)
)

一些考虑：

1. user + assistant 合并为一行，去掉 `role` 字段。一个 user 消息对应一个 assistant 回复，天然配对，行数减半，查询简单
2. ID 为 2 bytes，最大 65535 在AI上下文爆掉之前足够
3. 主键 = `(conversation_id, id)` 复合主键
4. parents 存所有祖先前缀路径，不含自身 ID，替代 parent_id + depth 这种需要递归的查询


## Consequences

优点：

- 一问一答一行，结构紧凑
- 2-byte ID 足够且紧凑
- parents 前缀查询高效（LIKE 前缀走索引）
- depth 无需额外字段
- 路径可直接用于面包屑导航
